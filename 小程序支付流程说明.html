小程序支付流程说明

微信小程序端付款结算页面的完整支付流程、抵扣链规则与场景示例。
版本 v1.0 · 2026-06-07

1. 概述

小程序端支付流程是用户扫描设备二维码后进入的付款结算页面。用户确认订单内容、查看会员资产消耗明细，然后发起支付。系统按固定优先级自动抵扣会员资产，不足部分通过微信支付补差。

核心原则：支付前必须可预览、可确认、可追溯。

2. 页面状态流转

小程序支付共有 5 个页面状态：

加载中 loading → 付款结算 confirm → 支付中 processing → 支付成功 success
支付中 processing → 支付失败 failed
状态说明
loading扫码后加载订单信息、查询会员数据
confirm展示订单内容 + 会员资产 + 支付明细，用户确认后发起支付
processing提交支付请求，展示处理中动画，禁止退出
success支付成功，展示支付方式、订单详情，指引用户前往设备体验
failed支付失败，展示失败原因和重试/返回入口

3. 抵扣链（核心计费规则）

系统按固定优先级自动计算每笔支付的资产消耗，顺序不可调整：

优惠券(减面值) → 会员折扣 → 预存款 → 游戏币 → 微信支付(补差)

3.1 详细计算步骤

步骤名称计算公式
①原价originalPrice = unitPrice
②优惠券后amountAfterCoupon = max(0, originalPrice - couponDiscountAmount)
③会员折扣后应付priceAfterDiscount = amountAfterCoupon × (member.discount / 10)
④预存款抵扣prepaidDeduction = min(member.balance, priceAfterDiscount)
⑤预存款抵扣后剩余afterPrepaid = priceAfterDiscount - prepaidDeduction
⑥游戏币抵扣量coinDeduction = min(member.coins, ceil(afterPrepaid))
⑦游戏币抵扣金额coinValue = coinDeduction（汇率 1:1）
⑧微信应付 / 缺口remainingPay = afterPrepaid - coinValue

3.2 关键规则

规则说明
汇率游戏币与人民币为 1:1
游戏币粒度不足 1 元时按 1 币扣除（向上取整）
不为负每一步结果均 max(0, ...) 确保不出现负值
缺口判定shortage = remainingPay，当 shortage > 0 时判定为预存款+游戏币不足

4. 付款结算页（confirm 状态）

4.1 页面组成

模块内容
导航栏返回按钮 + "付款结算"标题
会员卡片姓名、等级、折扣、预存款、游戏币、套票、预存次数
订单内容游戏名称、时长、标签、原价
支付明细完整抵扣链展示（原价 → 优惠券 → 折扣 → 抵扣 → 应付）
底部按钮确认并支付 / 微信补差

4.2 不足提醒

当 shortage > 0 时触发以下视觉变化：

  • 会员卡片变色（紫色渐变 → 深紫色）
  • 显示黄色警告条："预存款+游戏币不足，需微信补差"
  • 支付明细末尾"应付"金额变红
  • 底部按钮变红，文案改为"微信补差 ¥xx.xx"

4.3 按钮文案规则

条件按钮文案
shortage > 0微信补差 ¥xx.xx
shortage == 0确认并支付 ¥xx.xx
处理中loading 动画 + "处理中..."

5. 支付成功页（success 状态）

支付成功后展示三个模块：

5.1 支付方式标签

根据实际支付构成显示标签：

  • 有预存款抵扣 → 预存款 ¥xx.xx
  • 有游戏币抵扣 → xx 游戏币
  • 有微信支付 → 微信支付 ¥xx.xx

5.2 订单详情

字段内容
游戏游戏名称
时长xx分钟
订单号VR202606070001 格式

5.3 设备提示

"xx号设备 已准备就绪，请前往开始体验"

6. 支付失败页（failed 状态）

场景展示标题说明触发条件
订单已失效订单已失效订单超时或已被使用后端返回 order_status = expired/cancelled，如订单超过有效期、已被他人支付、设备端取消订单
支付超时支付超时支付处理超时点击确认支付后，微信支付 API 在 30~60 秒内无响应，前端请求超时或后端回调超时
余额不足余额不足微信支付余额不足微信支付 JSAPI 返回失败，用户微信零钱/银行卡余额不足以覆盖剩余应付金额

注意：failTitle 由调用方（父组件 / 后端 API 返回）通过 prop 传入，MiniPayFlow 组件不自行判断失败类型。

6.2 失败处理指引

  • 返回设备端重新展示最新付款码
  • 确认网络正常后再次发起支付
  • 如已扣款，稍后在订单中心查看结果

7. 场景示例

以下 8 种场景覆盖主流支付组合（原价 38 元，基础折扣 9 折）：

#场景优惠券折扣预存款游戏币折后应付预存抵扣币抵扣微信应付
1纯预存款全覆盖09100034.2034.2000
2纯游戏币全覆盖0905034.200350

7优惠券+折扣+全额覆盖58.51003028.0528.0500

8. 组件接口

8.1 Props（MiniPayFlow 组件）

属性类型说明
stateString页面状态：loading / confirm / processing / success / failed
memberObject会员信息（name, avatar, level, discount, balance, coins, points, tickets, times）
productObject商品信息（name, duration, tag, color, iconSvg）
unitPriceNumber折前原价
couponDiscountAmountNumber优惠券抵扣金额
failTitleString支付失败标题
orderNoString订单号
deviceLabelString设备标识，如 "03号设备"
storeNameString门店名称
qrStatusHintString扫码状态提示

8.2 Events

事件触发时机载荷
confirm用户点击确认支付{ scenario, prepaidDeduction, coinDeduction, remainingPay }
back点击返回-
cancel用户取消-
retry支付失败后重试-
done点击完成-

9. 与收银系统的差异

对比项小程序支付收银系统
支付方式选择不显示（仅微信支付）支持多种外部支付
会员资产选择系统自动计算，不可手动干预可配置优先级
适用场景用户自扫码支付店员代收银
游戏币显示作为抵扣明细展示不作为支付按钮

10. Demo 预览项目

小程序支付流程 Demo 位于 miniapp-payment/ 目录：

  • MiniPayFlow.vue — 支付流程核心组件，实现 5 种状态和抵扣计算

cd miniapp-payment && npx vite --port 9528
